# 規格文件管理

建立與管理功能規格文件，確保開發前有完整設計規劃。

## 使用方式

```
spec new {feature-name}      # 建立新規格書
spec update {feature-name}   # 更新現有規格
spec view {feature-name}     # 檢視規格
```

## 規格書位置

依功能領域分類存放，檔名帶功能名稱：

```
docs/
├── {domain-a}/{feature}_規格書.md
├── {domain-b}/{feature}_規格書.md
└── {domain-c}/{feature}_規格書.md
```

## 規格書狀態

| 狀態 | 說明 |
|------|------|
| 草稿 | 正在撰寫中 |
| 審查中 | 等待技術審查 |
| 已批准 | 可開始實作 |
| 已實作 | 功能已完成 |
| 已部署 | 已上線 |

## 規格書結構

1. **文件資訊** — 日期、版本、作者、狀態
2. **功能概述** — 背景、目標、範圍（含/不含）
3. **需求規格** — 功能/非功能需求、驗收標準
4. **技術設計** — 架構層級、資料模型、整合設計
5. **實作計劃** — 檔案清單、開發步驟
6. **測試計劃** — 單元/整合/效能測試
7. **風險評估** — 技術風險、已知限制
8. **變更歷史** — 版本記錄

## 規格書模板

以下模板可直接落檔後填寫，層名與整合面向依專案實際調整：

```markdown
# {功能名稱} 規格書

## 文件資訊
- **建立日期**: {日期}
- **最後更新**: {日期}
- **版本**: 1.0.0
- **作者**: {作者}
- **狀態**: 草稿 | 審查中 | 已批准 | 已實作

## 1. 功能概述
### 1.1 背景
{功能的業務背景與目的}
### 1.2 目標
{此功能要達成的目標}
### 1.3 範圍
- 包含: {功能包含的內容}
- 不包含: {功能不包含的內容}

## 2. 需求規格
### 2.1 功能需求
1. 系統應該...
2. 使用者可以...
3. 當...時，系統必須...
### 2.2 非功能需求
- **效能**: {回應時間 / 吞吐量目標}
- **可用性**: {上線時間目標}
- **可擴展性**: {擴展策略}
### 2.3 驗收標準
- [ ] 標準 1
- [ ] 標準 2

## 3. 技術設計
### 3.1 架構層級
- **Domain 層**: {實體、值物件、介面}
- **Application 層**: {服務、Commands/Queries、DTOs}
- **Infrastructure 層**: {Repository 實作、外部整合}
- **Presentation 層**: {API 端點、背景工作}
### 3.2 資料模型
{資料表 / 文件結構設計}
### 3.3 整合設計
{外部 API / 訊息 / 背景排程的端點、格式、處理流程、錯誤處理}

## 4. 實作計劃
### 4.1 檔案清單
- [ ] Domain/Entities/{Entity}
- [ ] Application/Services/{Service}
- [ ] Infrastructure/Repositories/{Repository}
- [ ] Tests/{Test}
### 4.2 開發步驟
1. 建立 Domain 實體
2. 實作 Application 服務
3. 建立 Repository
4. 撰寫並執行測試

## 5. 測試計劃
### 5.1 單元測試
{測試案例列表、覆蓋率目標}
### 5.2 整合測試
{整合測試場景、驗證項目}
### 5.3 效能測試
{效能指標、負載測試}

## 6. 風險評估
### 6.1 技術風險
- 風險 1: 描述及緩解措施
### 6.2 已知限制
- 限制 1

## 7. 變更歷史
| 版本 | 日期 | 作者 | 變更內容 |
|------|------|------|---------|
| 1.0.0 | {日期} | {作者} | 初始版本 |
```

## 品質檢查

規格文件應包含：

- [ ] 清晰的功能描述
- [ ] 詳細的技術設計
- [ ] 明確的驗收標準
- [ ] 實作計劃
- [ ] 測試計劃
- [ ] 風險評估

## 與其他 skill 銜接

- `review` — 審查規格文件
- `work-on` — 依規格執行開發
- `commit` — 提交時引用規格文件
