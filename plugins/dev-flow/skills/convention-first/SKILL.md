# Convention-First Workflow

> **核心信念**：成熟的 codebase 已有大量既有實作。新增任何元件前，**必須**先模仿既有模式，禁止憑感覺引入新風格。

## 為什麼

- 一致性 > 個人偏好，降低長期維護成本。
- 既有模式已通過 review、有測試覆蓋、團隊熟悉。
- 引入新模式 = 增加維護負擔 + Code Review 阻力 + 團隊學習成本。

## 三步流程

### Step 1：搜尋既有實作

依你要寫的元件類型，先 grep 出同類既有實作當樣板。範例（依專案語言與命名慣例調整關鍵字）：

| 我要寫的元件 | 搜尋策略 |
|------------|---------|
| 某類 handler / service | grep 同類命名（如 `*Handler` / `*Service`），限縮到相近模組目錄 |
| 資料存取層 | grep 既有的 repository / DAO / query 實作 |
| 設定 / 組態 | grep 既有的 config / options / 設定類別 |
| 驗證器 | grep 既有 validator 的基底型別或介面 |
| 控制器 / 路由 | grep 既有 controller / route handler 的基底類別 |
| 資料轉換 / 映射 | grep 既有 mapper / DTO 轉換 |

要點：**先找同類、限縮到相近模組目錄、看出現頻率最高的主流寫法**。

### Step 2：依搜尋結果判斷

| 找到 | 行動 |
|------|------|
| **≥2 個同類實作** | 模仿主流寫法（出現頻率最高的） |
| **1 個** | 模仿，但先確認它是否合理（查 git log 看是近期新模式還是歷史包袱） |
| **0 個** | **觸發反問** → 跳到 Step 3，不可自行引入新風格 |

### Step 3：找不到既有實作時的反問

**不可**自行引入新風格。反問格式：

```
[需要決策] 找不到 {元件類型} 的既有樣板
- 我打算引入：{方案 A 描述}
- 替代方案：{方案 B}（如果有）
- 為什麼需要拍板：{會影響後續同類元件的標準}
請決定，或補充我遺漏的搜尋方向。
```

反問對象：
- 開發角色 → 向架構決策者或使用者確認（這是新需求類型時尤其要問）。
- 審查角色 → 在審查報告標註「找不到既有樣板，建議由架構決策者拍板新模式」。

## 禁止行為

| 反例 | 為何錯 |
|------|-------|
| 「我覺得某 pattern 比較好用」單方面引入 | 違反慣例——既有已有一套錯誤/結果處理慣例 |
| 自寫一套抽象替代專案既有的框架用法 | 增加學習成本，與專案慣例衝突 |
| 沒搜尋就「依直覺」寫元件 | 一定會偏離專案命名 / 結構慣例 |
| reviewer 要求「重寫成更簡潔的寫法」但專案普遍是另一種風格 | 審查偏差——個人偏好強加 |

## 與 reviewer 的關係

reviewer 載入此 skill 後，審查時**反向應用**：

- 看到「新風格」程式碼 → 反問「是否有既有樣板？為何不沿用？」
- 看到「跟既有樣板一致」→ 給予正面評價。
- **禁止**：reviewer 自己提出新模式建議（除非原始樣板有明顯缺陷）。

## 例外情境

容許脫離慣例的條件（須在 PR 描述明確說明）：

1. 既有樣板有明確 bug 或 anti-pattern（例：違反基本設計原則）。
2. 新需求類型在專案中沒有任何先例（須先反問架構決策者）。
3. 既有樣板已被標為 deprecated（git log / PR 中可查證）。
4. 安全或效能重大缺陷，無法漸進改善。

任一條件成立 → 在 PR 描述加 `## Convention Deviation` 段落說明理由。
