# 程式碼審查

委派專業審查型 subagent，對變更做全面的架構與品質審查。

## 審查範圍

| 範圍 | 用法 | 說明 |
|------|------|------|
| PR 審查 | `review pr {n}` | 審查特定 Pull Request |
| 本地變更 | `review changes` | 審查未提交變更 |
| 分支審查 | `review branch feature/xxx` | 審查特定分支 |

## 工作流程

1. **準備** — 檢查 `git status` / `git diff`；若審 PR 則 `gh pr checkout {n}`。
2. **品質檢查** — 先跑建置與測試確認基本盤（見 `quality-check`）。
3. **委派審查** — 交給審查型 subagent，聚焦下列面向：
   1. Clean Architecture 分層依賴與依賴方向
   2. 架構決策合理性
   3. 外部整合／背景排程／資料存取的實作品質（依專案實際採用者）
   4. 測試覆蓋率與測試品質
   5. 錯誤處理、效能、安全性、可擴展性
4. **彙整報告** — 合併結果，給出總體評估與行動項目。

## 審查檢查清單

### Clean Architecture
- [ ] Domain 層無外部依賴
- [ ] 層級依賴方向正確（外層依賴內層）
- [ ] 介面定義在適當的層
- [ ] 依賴注入正確使用

### 資料存取
- [ ] 無 N+1 查詢
- [ ] 查詢已最佳化、索引適當
- [ ] 連線／交易管理正確

### 整合與背景工作（如適用）
- [ ] 連線／重試／逾時管理完整
- [ ] 背景工作支援取消令牌
- [ ] 錯誤處理與日誌記錄充分

### 測試
- [ ] 單元測試覆蓋新功能
- [ ] 測試案例有意義、涵蓋邊緣案例
- [ ] 測試可維護

### 安全性
- [ ] 輸入驗證完整
- [ ] 無敏感資訊暴露
- [ ] 認證授權正確
- [ ] 錯誤訊息不洩漏內部資訊

### 效能
- [ ] 無不必要的查詢
- [ ] 適當使用批次處理
- [ ] 非同步操作正確、無阻塞
- [ ] 無記憶體洩漏風險

## 報告格式

```markdown
# 程式碼審查報告

## 整體評估
[✅ 核准 / ⚠️ 需要修改 / 🚫 要求變更]

## 品質指標
| 項目 | 分數 |
|------|------|
| Clean Architecture | [/10] |
| 資料存取 | [/10] |
| 測試覆蓋率 | [%] |
| 效能 | [/10] |
| 安全性 | [/10] |

## 架構評估
[分層與依賴方向的分析]

## 發現的問題
[問題 + 建議]

## 行動項目
- [ ] [需修正項目 1]
- [ ] [需修正項目 2]
```

## 審查結果分類

- ✅ **Approved（核准）** — 無重大問題，可合併。
- ⚠️ **Needs Changes（需要修改）** — 有次要問題，不阻擋合併但建議修正。
- 🚫 **Request Changes（要求變更）** — 有重大問題必須修正，阻擋合併。
